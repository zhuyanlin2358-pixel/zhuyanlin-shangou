## 当前合作方式

### 1. 需求反馈（所有人）
通过 [需求漏斗](./02-需求漏斗.md) 按格式提交。  
不接受口头需求，不接受无背景的建议。

### 2. 模板贡献 · 业务/运营
在工具中搭建好的活动页面，可沉淀为模板供他人复用：

1. 在工具中完成页面配置
2. 点击「保存为模板」导出 `.template.json`
3. 发给朱彦霖审核（钉钉/微信）
4. 审核通过 → 加入模板库，署名贡献者，**每次被复用时收到通知**

审核标准：符合视觉规范 · 有明确适用场景 · 在真实活动中验证过

### 3. 模板贡献 · 设计师
设计师用 Figma 或配置表单创作，**不需要了解任何代码**。  
→ 详见 [设计师协作指南](./04-设计师协作指南.md)

### 4. 技术讨论（有开发背景）
欢迎对技术方案（Canvas 渲染、导出逻辑、组件架构）提出想法。  
最终决策权在项目负责人，以保证代码库一致性。

## 技术架构参考：设计 → 工程的桥接方式

> 以下供有技术背景的合作方参考，业务/设计师无需阅读。

### 核心接口：`.template.json` 格式

设计师的产出（Figma 导出或表单生成）最终统一为以下结构：

```json
{
  "version": "1.0",
  "templateName": "618主会场",
  "author": "设计师姓名",
  "pageStyle": { "backgroundColor": "#fff", "width": 375, "height": 667 },
  "components": [
    {
      "type": "HeroBanner",
      "props": {
        "title": "狂欢618",
        "backgroundImage": "https://xxx.jpg",
        "ctaText": "立即抢购"
      }
    },
    {
      "type": "SlotMachine",
      "props": { "colorKey": "red", "titleText": "惊喜抽奖" }
    }
  ]
}
```

**关键原则：JSON 只描述"用了什么组件 + 配置是什么"，不包含任何样式代码。样式由 React 组件库保证。**

### 模板加载器：ComponentMap 动态渲染

```typescript
// src/utils/templateLoader.ts
import { SlotMachine } from '@/components/pages/SlotPage'
import { FloorBar }    from '@/components/pages/FloorPage'
import { HeroBanner }  from '@/components/ui/HeroBanner'
// ... 其他组件

const ComponentMap: Record<string, React.ComponentType<any>> = {
  'SlotMachine': SlotMachine,
  'FloorBar':    FloorBar,
  'HeroBanner':  HeroBanner,
}

export function renderTemplate(template: PageTemplate) {
  return template.components.map(item => {
    const Component = ComponentMap[item.type]
    if (!Component) { console.warn(`未知组件类型: ${item.type}`); return null }
    return <Component key={item.id} {...item.props} />
  })
}
```

新增组件只需：① 开发 React 组件 → ② 加入 `ComponentMap` → ③ 更新 Figma 组件库。

### 保存模板核心代码

```typescript
// src/utils/template.ts
export interface PageTemplate {
  version: '1.0'
  templateName: string
  author: string
  createdAt: string
  pageStyle: { backgroundColor: string; width: number; height: number }
  components: Array<{ type: string; props: Record<string, unknown>; id?: string }>
}

export function downloadTemplate(template: PageTemplate) {
  const blob = new Blob([JSON.stringify(template, null, 2)], { type: 'application/json' })
  const a = Object.assign(document.createElement('a'), {
    href: URL.createObjectURL(blob),
    download: `${template.templateName.replace(/\s+/g, '-')}.template.json`,
  })
  document.body.appendChild(a); a.click(); a.remove()
  URL.revokeObjectURL(a.href)
}
```

## 工具的护城河：不是功能，是积累

竞争者可以复制功能，但复制不了三件事：

**① 历史配置数据**  
每一次大促、每一套配色、每一个高复用模板都沉淀在工具里。"系统知道上次618哪套配色转化最好"——新工具从零开始无法拥有。

**② 组织最佳实践记忆**  
成功的活动配置自动成为候选模板，工具用得越久推荐越精准。当某个会场配置复用超过阈值，系统自动推送"这个值得固化为模板"——无需任何人主动操作。

**③ 判断力**  
Figma 设计 → JSON → React 的转换过程，不只是技术翻译，而是品牌合规性把关。同时理解"业务要什么"和"技术能做什么"的判断力，无法被代码替代。

**对外话术：**
> "我们为各位的创作成果做**质检、包装和快递**。你用别的工具重新做，数据从零；用我们的，系统知道过去所有大促哪套方案最有效。"

## 为什么暂时封闭？

- **快速迭代需要**：早期频繁调整架构，多人协作会引入冲突和技术债
- **规范尚在建立**：技术栈和组件规范仍在完善中
- **首要目标**：让工具稳定解决真实业务问题，而不是建立完美协作流程

> **先把工具做"对"，再把它做"大"。**
