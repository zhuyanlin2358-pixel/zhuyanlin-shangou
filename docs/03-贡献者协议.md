## 当前合作方式（四种）

### 1. 需求反馈
通过 [需求漏斗](./02-需求漏斗.md) 按格式提交。  
不接受口头需求，不接受无背景的建议。

### 2. 模板贡献（主动提交）
如果你在工具中搭建了一个效果好的活动页面，可以将其沉淀为模板：

1. 在工具中完成页面配置
2. 使用「保存为模板」功能导出 `.template.json` 文件
3. 将文件发给朱彦霖审核
4. 审核通过后加入官方模板库，**署名贡献者，并在每次被复用时收到通知**

**模板审核标准：**
- 符合美团闪购视觉规范
- 有明确的适用场景（如"大促主场 - 热烈红"）
- 在真实活动中验证使用过

> 💡 **亦仁补充**：你不需要主动提交才能沉淀模板。当某个会场配置被多人复用、或对应活动下载量超过阈值，系统会自动向维护人推送"这个配置值得固化为模板"的提示。**好的设计自然浮现，无需依赖任何人主动操作。**

### 3. UI 模块贡献（模块实验室 · HTML 提交）

适用场景：你（或你的前端同学）有一个 UI 想法，但不熟悉 React/TypeScript。

**你的工作：写一个自包含的 `.html` 文件**

要求：
- 完整 HTML 结构 + 内联样式（可引用 CDN CSS）
- 用注释标明可配置参数：

```html
<!-- 可配置: data-color="primary|success|danger" -->
<!-- 可配置: data-label="按钮文字" -->
<button
  data-color="primary"
  data-label="立即领取"
  style="background:#fad900; color:#1a0a00; padding:12px 32px; border-radius:999px; font-weight:700; border:none; cursor:pointer;"
>
  立即领取
</button>
```

- 在 Chrome 最新版本测试通过
- 文件完全自包含（不依赖本地路径）

**提交方式：** 将 `.html` 文件发给朱彦霖（微信/钉钉）

**朱彦霖做什么：**  
用 AI（Cursor）将你的 HTML 解析、重构为符合项目规范的 React 函数组件，处理好 Props 接口，并集成进工具。**这个步骤对你完全透明，你不需要了解 React。**

转换示例：
```typescript
// 你提交的 HTML → 自动转换为：
interface CouponButtonProps {
  color?: 'primary' | 'success' | 'danger'
  label?: string
  onClick?: () => void
}

export function CouponButton({ color = 'primary', label = '立即领取', onClick }: CouponButtonProps) {
  const bgMap = { primary: '#fad900', success: '#3ac749', danger: '#ff3e58' }
  return (
    <button onClick={onClick} style={{ background: bgMap[color], color: '#1a0a00', padding: '12px 32px', borderRadius: 999, fontWeight: 700, border: 'none', cursor: 'pointer' }}>
      {label}
    </button>
  )
}
```

## 模板保存 · 技术实现参考

以下为「保存为模板」功能的核心代码结构，供技术同学参考：

```typescript
// src/utils/template.ts

export interface PageTemplate {
  version: '1.0'
  name: string
  description?: string
  author: string
  createdAt: string
  pageStyle: {
    backgroundColor: string
    width: number
    height: number
  }
  components: ComponentConfig[]
}

/** 从当前画布状态导出模板 JSON */
export function exportAsTemplate(
  name: string,
  author: string,
  components: ComponentConfig[],
  pageStyle: PageTemplate['pageStyle']
): PageTemplate {
  return { version: '1.0', name, author, createdAt: new Date().toISOString(), pageStyle, components }
}

/** 触发浏览器下载 .template.json */
export function downloadTemplate(template: PageTemplate) {
  const blob = new Blob([JSON.stringify(template, null, 2)], { type: 'application/json' })
  const a = Object.assign(document.createElement('a'), {
    href: URL.createObjectURL(blob),
    download: `${template.name.replace(/\s+/g, '-')}.template.json`,
  })
  document.body.appendChild(a)
  a.click()
  a.remove()
  URL.revokeObjectURL(a.href)
}
```

## 工具的护城河：不是功能，是数据积累

> 以下来自亦仁的建议，记录在此作为产品方向参考。

**为什么这个工具很难被复制？**

竞争者可以复制功能，但复制不了三件事：

1. **历史配置数据**  
   每一次大促、每一套配色方案、每一个下载量超过 N 次的模板，都沉淀在工具里。  
   "系统知道上次618这个配色搭配转化最好" —— 这是新工具从零开始无法拥有的。

2. **组织最佳实践记忆**  
   成功的活动页面自动成为候选模板，失败的配置不会再被推荐。  
   工具用得越久，推荐越精准，学习曲线越短。

3. **你的判断力**  
   HTML → React 的转换过程，不只是技术翻译，而是品牌合规性把关。  
   你是唯一同时理解"业务要什么"和"技术能做什么"的人，这个判断力无法被代码替代。

**对外的话术（给不熟悉技术的领导/合作方）：**

> "我们不是来增加大家工作的，我们是为各位的创作成果做**质检、包装和快递**的——  
> 确保大家的辛苦设计能 100% 无损地高效上线。  
> 你用别的工具重新做，数据从零开始；用我们的，系统知道过去所有大促哪套方案最有效。"
