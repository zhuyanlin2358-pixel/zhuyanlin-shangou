# 📊 数据仪表盘 · 增长引擎

> 数据仪表盘是整套治理系统的**反馈回路**。  
> 没有它，你无法证明工具的价值，无法判断迭代方向，也无法在组织里争取资源。  
> 它不是技术装饰，是系统能否"自运转"的根基。

---

## 为什么必须有？

| 场景 | 没有数据时 | 有数据时 |
|------|-----------|---------|
| 向领导汇报 | "我觉得工具挺有用的" | "上月47个活动用了工具，节省约210人时" |
| 争取开发资源 | "这个功能应该有用" | "模板功能被调用128次，是最高频操作" |
| 决定迭代方向 | 凭感觉猜 | 看用户在哪步流失，哪个功能没人用 |
| 建立护城河 | 别人能复制功能 | 无法复制你积累的真实使用数据 |

---

## 追踪什么？（三级指标）

### 一级：给领导看的（每月报一次）

- **月活跃用户数**：有多少人在真正使用
- **活动产出数**：每月用工具完成了多少个活动页面
- **效率提升**：平均每个活动节省多少小时
- **成本节约**：折算成人力成本，省了多少钱

### 二级：给自己看的（每周看一次）

- **各功能使用频率**：哪些功能最常用，哪些无人问津
- **模板使用排行**：哪个模板最受欢迎
- **用户留存**：新用户第二周还会回来吗
- **流失节点**：用户在哪个步骤放弃

### 三级：深度分析用的（按需）

- **用户行为路径**：从进入到完成，走了几步
- **需求来源分布**：哪个业务线贡献的需求最多
- **贡献者网络**：谁提交的模板被复用最多次

## 如何实现？（三个阶段）

### 阶段一：今天就能做——手动埋点

在工具关键操作节点加日志，5 行代码：

```typescript
// src/utils/analytics.ts
export function logEvent(event: string, data?: Record<string, unknown>) {
  const payload = { event, ts: Date.now(), ...data }
  // 开发阶段先打印到控制台，后续换成接口调用
  console.info('[analytics]', payload)
  // fetch('/api/log', { method: 'POST', body: JSON.stringify(payload) })
}

// 使用：在关键操作时调用
// logEvent('template_used', { name: '老虎机', colorKey: 'red' })
// logEvent('asset_downloaded', { comp: 'slot', count: 8 })
// logEvent('page_published')
```

**埋点位置（优先级顺序）：**
1. 「完成·下载素材」点击 → `asset_downloaded`
2. 「加入会场」点击 → `comp_added`
3. 「保存为模板」点击 → `template_saved`
4. 用户进入各页面 → `page_view`

### 阶段二：一周内——本地统计表

暂时不做后端，用飞书多维表格手动记录：

| 日期 | 活动名称 | 使用组件 | 下载素材数 | 操作人 | 耗时（分钟）|
|------|---------|---------|-----------|--------|-----------|
| 06-25 | 618主会场 | 老虎机+楼层条 | 12 | 运营小王 | 28 |

每周五花 10 分钟汇总。**这张表是你最初始的「数据护城河」。**

### 阶段三：一个月内——可视化仪表盘

在工具里加 `/admin/dashboard` 路由（仅你可见），展示：


## 反馈飞轮：数据如何连接四个模块

```
01 使用指南
  └─ 文末：「遇到问题？点此反馈」→ 直达 02 需求漏斗
  
02 需求漏斗  
  └─ 需求关闭时：自动 @ 提出者「已上线，预计节省你 Z 小时」
  
04 设计师协作
  └─ 模板画廊每个模板显示「已被使用 N 次，共节省 M 小时」
  
05 数据仪表盘（汇总所有数据）
  └─ 每周输出一页简报：「本周为大家节省约 XXX 小时」
  └─ 发大群 / 发邮件组 / 贴在汇报 PPT 里
```

**这个飞轮让价值可见、贡献可见，形成正向增强回路。**

## 对外话术（用数据讲故事）

**对业务/运营说：**
> "用这个工具，你上次那个活动 28 分钟就完成了，上上次走旧流程是 2 天。这是我们记录的数据。"

**对领导说：**
> "上个月 47 个活动用了这套工具，节省了约 210 人时，折合人力成本约 XX 元。这是持续在跑的数字，不是估算。"

**对想复制你的人：**
> "功能可以复制，但我们积累的 6 个月使用数据、哪套配色在大促转化最好、哪个模板被复用最多——这些复制不了。"

## 压力测试：四个问题

在你认为系统搭好之前，问自己：

1. **你休假两周，系统能运转吗？**  
   01 能自学 → 02 能积压等你 → 04 不阻塞设计师 → 05 数据还在跑

2. **业务量翻倍，哪里会崩？**  
   02 漏斗堵了 → 加过滤规则；04 接口慢了 → 推严格 Figma 规范

3. **有人强行塞不合理需求，怎么防？**  
   02 强制格式是第一道墙；05 数据是谈判依据（"这个需求投入产出比低于平均水平"）

4. **怎么让用工具变成一件「有面子」的事？**  
   05 模板画廊公开复用次数 + 贡献者署名 = 可见的荣誉和影响力
